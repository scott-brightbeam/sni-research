# Helsinn Phase 1 Proposal — Claude Code instructions

You are working inside `web/helsinn/`, a self-contained sub-app of the SNI Research v2 repo. **The Helsinn site is independent of SNI** — it has its own Fly app, its own server, its own SPA bundle. The only thing it shares with SNI is the Turso database (with strictly namespaced `helsinn_*` tables) and the `.env` file at the repo root.

Always read `web/helsinn/HANDOVER.md` first. It's the canonical narrative.

## Hard rules

1. **Never run `bun test` or any pipeline command from the SNI root that hits the Anthropic API.** SNI's cost-protection rule applies in this subtree too.
2. **Never edit the visual structure of the JSX components or rename CSS classes.** The styles in `src/styles.css` depend on the existing class names. Copy edits inside JSX are fine; structural changes are not.
3. **Never commit secret values to git.** Anything in `process.env.*` lives in Fly secrets, not in code. Use `fly secrets set --app helsinn-proposal-1` and verify with `fly secrets list`.
4. **Don't change the login or admin-login page visuals without explicit user sign-off.** Both have been visually approved by Scott. Copy edits only.
5. **Don't add third-party scripts, fonts, analytics, or trackers to any HTML page.** The site is anti-indexing by design — `X-Robots-Tag`, `noindex` meta, no `robots.txt`, UA-blocklist for crawlers. Adding an external beacon breaks the design.
6. **Don't touch `src/search-index.jsx` unless the user asks.** It's a hand-curated corpus. Drift is an accepted trade-off vs the work of regenerating.

## Auth model — read carefully

Two parallel paths share one cookie (`hp_session`, JWT, 24 h):

| Path | Login URL | Endpoint | Sets `via=` | Admin? |
|---|---|---|---|---|
| Reviewer | `/login` | `POST /auth/login` (email+password) | `password` | **Never** |
| Admin (Brightbeam) | `/admin/login` | `GET /auth/google/start` → callback | `google` | If `email.endsWith('@brightbeam.com')` |

`isAdmin(payload)` (`server/server.js`) is the single gate. Admin requires `via='google'` AND `@brightbeam.com`. A self-claimed email via the password path is never trusted.

The auth gate middleware:
- `/auth/*` → public
- `PUBLIC_PATHS` (login pages, favicons, manifest, logo) → public
- `/admin/*` unauth → redirect to `/admin/login`
- everything else unauth → redirect to `/login`
- `/admin/analytics` non-admin (e.g. password user) → 403 with the "Brightbeam staff only" denial page (which has a "Switch to Google sign-in →" button carrying `?next=/admin/analytics`)

## Analytics

- Schema: `helsinn_users`, `helsinn_sessions`, `helsinn_views` in Turso. `ensureSchema()` runs on every server start (idempotent `CREATE TABLE IF NOT EXISTS`).
- Client: `src/analytics.js`. `installAnalytics()` once at App mount, `trackView({section, tab, anchor})` on every route change. Dwell is wall-clock minus hidden time; closed via `navigator.sendBeacon` on pagehide.
- Server endpoints: `POST /a/view`, `POST /a/heartbeat`, `POST /a/end`. Auth-gated by middleware. Server reads `email` + `sid` from JWT — never trusts client-supplied identity.
- Dashboard: `GET /admin/analytics` (HTML, Google admin gated). JSON: `GET /api/admin/summary`. Renderer in `server/dashboard.js`.

To wipe analytics for a clean slate:
```bash
set -a && source /Users/scott/Projects/sni-research-v2/.env && set +a

```
